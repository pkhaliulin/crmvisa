Текущий статус
Проект находится на стадии controlled production pilot. Основной бэкенд, CRM-интерфейс, клиентский портал и Owner-панель функционируют. Однако SMS-верификация работает через заглушку (фиксированный PIN), платежи принимаются вручную (без webhook-интеграции с платёжными системами), OCR документов отсутствует. Это означает, что система пригодна для ограниченного пилотного запуска с контролируемым набором агентств и сценариев, но не для полноценного GA (General Availability) без доработки критичных интеграций.

Честное замечание о RLS
RLS — мощный защитный механизм, но его эффективность зависит от корректной настройки: запрет bypassrls для application-роли, отсутствие superuser-подключений из приложения, правильная установка session context. В текущем CI RLS не тестируется (тесты на SQLite), поэтому корректность RLS-политик подтверждена только ручной проверкой на production PostgreSQL, а не автоматически.

Рекомендация: добавить отдельный integration test suite на реальном PostgreSQL с проверкой RLS, session context, raw SQL access и отрицательными кейсами (попытка доступа к чужим данным через raw query).

Все маршруты живут под /api/v1/, но формальной стратегии версионирования нет. При едином бэкенде для B2C, B2B и Owner Panel без контрактного управления изменение API для одного интерфейса может сломать другой. OpenAPI-спецификация ещё не создана. При росте команды это станет болезненным.

Про "независимость" модулей
Модули организованы как доменные папки с собственными Models, Controllers, Services, Events, Policies, Resources, Requests. Это хорошо как организационная структура кода и значительно лучше, чем монолитная свалка.

Однако это не доказанные bounded contexts с контрактными зависимостями и слабым сцеплением. Модули ссылаются друг на друга напрямую (Service одного модуля инжектит Model другого). Правильнее называть это "доменно-организованный код", а не "независимые модули". Для настоящей модульной архитектуры нужны: явные публичные API модулей, запрет прямых cross-module зависимостей, контрактные интерфейсы и event-driven communication между bounded contexts.

Инфраструктура тестов

  • Engine: PHPUnit + SQLite in-memory (быстро: ~3.5 секунды)
  • Фабрики: database/factories/Modules/{Agency,User,Client,Case,Payment}/
  • Helpers: createAgencyWithOwner(), createManager(), createClient(), authHeaders(), actingAsUser()
  • CI: GitHub Actions — автозапуск при push/PR
Ключевое ограничение: SQLite vs PostgreSQL

Все 159 тестов выполняются на SQLite in-memory. PostgreSQL-специфичный SQL обёрнут в driver-check условия, RLS-политики и SET LOCAL обходятся условием if (driver !== 'sqlite').

Для обычного Laravel CRUD это приемлемый trade-off (скорость vs точность). Но для системы, где один из ключевых security pillars — именно PostgreSQL RLS, это означает, что главный защитный механизм не тестируется автоматически.

Рекомендация:

  • Добавить отдельный integration test suite на реальном PostgreSQL (можно в Docker)
  • Тесты: включённый RLS, проверка session context, raw SQL access, отрицательные кейсы (попытка доступа к чужим данным)
  • Запускать как отдельный job в CI (не блокирует основной suite)
Что не покрыто тестами
  • E2E тесты (browser-based)
  • Нагрузочное тестирование
  • PostgreSQL RLS integration tests
  • Contract tests для API (нет OpenAPI spec)
  • Concurrency tests для биллинга (double-charge, race conditions)

Отсутствие метрик и бенчмарков

Перечисленные меры — это правильные архитектурные решения, но их эффективность не доказана измерениями.

На данный момент отсутствуют:

  • Baseline latency (p50/p95/p99) для ключевых эндпоинтов
  • Cache hit ratio (какой % запросов реально обслуживается из кэша)
  • CPU/RAM/IO профиль под нагрузкой
  • Результаты нагрузочного тестирования (k6, artillery, или хотя бы ab)
  • Сравнение warm cache vs cold cache
  • Benchmark read-heavy vs report-heavy сценариев

Утверждения о конкретных процентах снижения нагрузки или миллисекундах ответа невозможно сделать без этих данных. Архитектура спроектирована с запасом, но доказательство — только нагрузочное тестирование.

Возможности биллинга

  • Upgrade / Downgrade: pro-rata credit при upgrade, отложенный downgrade (в конце периода)
  • Earn-first модель (Enterprise): платформа зарабатывает % от платежей клиентов агентства
  • Wallet / Ledger: внутренний баланс, все операции записываются в ledger
  • Купоны: процентные и фиксированные скидки
  • Аддоны: дополнительные менеджеры, кейсы сверх лимита
  • НДС: расчёт по стране агентства
  • Invoices: автоматическая генерация при подписке
  • Expired subscriptions: фоновый job (ежедневно 01:00)
  • Pending downgrades: фоновый job (ежедневно 00:05)
Открытые вопросы биллинга

Архитектура чистая, тесты покрывают основные сценарии (21 тест). Но для финансовой системы под боем есть критичные пробелы:

ВопросСтатусРиск
Webhook-интеграция (Stripe/Payme) Нет Платежи принимаются вручную. Автоматическое подтверждение оплаты невозможно
Идемпотентность обработки событий Не реализовано Повторный webhook может привести к double-charge
Reconciliation Нет Нет механизма сверки внутреннего ledger с платёжной системой
Unique constraints на external payment IDs Не проверено Защита от дублей на уровне БД

Concurrency tests Нет Race conditions при одновременных операциях на одном wallet
Accounting invariant tests Частично Есть тесты на баланс, но нет проверки "сумма всех операций = текущий баланс" как инвариант

Ограничения health-check

Текущий /api/health проверяет только liveness (DB подключение, Redis ping, disk space). Это полезно для UptimeRobot/Pingdom, но недостаточно для оценки реального состояния бизнес-логики.

Рекомендации:

  • Synthetic transaction: создание и удаление тестовой записи для проверки полного pipeline

11. Масштабируемость: гипотезы и необходимые проверки

Целевые параметры

ПараметрЦелевое значение
Агентств200
Заявок в год50 000
Одновременных пользователей~200-400
Документов~150 000 (3 на заявку)

Архитектурные решения, работающие на масштаб

  • Redis-кэш: снижает нагрузку на БД для read-heavy дашбордов (TTL 2-5 мин)
  • Composite индексы: 7 индексов на часто фильтруемые комбинации (agency+stage+status и др.)
  • Background jobs: SLA-проверки и billing не блокируют HTTP-запросы
  • Пагинация: все list-endpoints возвращают постраничные результаты
  • Eager loading: предотвращает N+1 проблемы
  • Tenant scoping: каждый запрос фильтрован по agency_id — объём данных в query пропорционален размеру агентства, а не всей платформы
Честная оценка

Архитектура спроектирована с запасом для заявленных параметров. Принятые решения (кэш, индексы, очереди, tenant scoping) — это стандартные и проверенные подходы для подобных нагрузок.

Однако это гипотеза, а не доказанный факт. Для подтверждения необходимо:

  • Нагрузочное тестирование (k6/artillery) с реалистичными сценариями
  • Профилирование CPU/RAM/IO под нагрузкой
  • Измерение p95/p99 latency для ключевых эндпоинтов
  • Проверка поведения при 200 агентствах с реальным объёмом данных
  • Тестирование cache hit ratio и warm/cold cache сценариев

Вероятный сценарий: при росте потребуются read replica для аналитических запросов, S3/CDN для документов, materialized views для тяжёлых отчётов, отдельный queue server. Это нормально — но означает, что "без изменений инфраструктуры" — слишком категоричное утверждение.

13. Disaster Recovery: текущий пробел

Состояние: не задокументировано

Для SaaS с документами и платежами disaster recovery — обязательный раздел. На данный момент:

ПараметрТекущее состояниеРекомендация
RPO (Recovery Point Objective) Не определён Установить RPO ≤ 1 час для данных, ≤ 24 часа для документов
RTO (Recovery Time Objective) Не определён Установить RTO ≤ 4 часа
Database backups Вероятно есть (VPS provider) Автоматический pg_dump + encrypted offsite backup (S3)
Backup verification Нет Ежемесячный тест восстановления из backup
Rollback strategy git revert + deploy Blue/green или canary deploy, database migration rollback plan
Runbooks Нет Документация: что делать при падении DB, Redis, disk full, certificate expiry
Документы (файлы) Только на диске сервера S3 + cross-region replication

15. Итоговый вердикт

Что сделано хорошо

  • Система перешла от "контроллер на стероидах" к зрелой сервисной архитектуре
  • Двухуровневая tenant isolation (Eloquent scope + PostgreSQL RLS) — серьёзный defence-in-depth
  • BillingEngine рефакторирован из монолита в facade с 4 специализированными сервисами
  • 4 Policy класса обеспечивают гранулярную авторизацию owner/manager
  • 159 тестов покрывают ключевые бизнес-сценарии и запускаются в CI
  • Event-driven архитектура с 11 событиями и фоновыми задачами
  • Structured logging по доменным каналам (auth, billing, slow queries)
  • Redis-кэш с event-driven инвалидацией для read-heavy операций
  • Все 7 выделенных блоков архитектурного долга закрыты

Что требует внимания до GA

  • SMS и платежи — заглушки, критично для полноценного запуска
  • RLS не тестируется в CI — главный защитный механизм не подтверждён автоматически
  • Отсутствуют нагрузочные тесты — масштабируемость остаётся гипотезой
  • Нет at-rest encryption для паспортных данных
  • Disaster recovery не задокументирован
  • OpenAPI и API versioning отсутствуют
  • OCR документов — core workflow для визового продукта, пока ручная обработка

Формулировка статуса

Проект готов к controlled production pilot — ограниченному пилотному запуску с контролируемым набором агентств и сценариев. Инженерная база сильная: чистая архитектура, тестовое покрытие, мультитенантная изоляция, авторизация на уровне сущностей, event-driven обработка и observability.

Для перехода к General Availability необходимы: реальные SMS и платёжные интеграции, PostgreSQL integration tests, at-rest encryption для PII, disaster recovery план и нагрузочное подтверждение.

Закрытие 7 блоков архитектурного долга — значительный шаг. Но архитектурный долг — это не чекбокс, а хроническое состояние, которое требует постоянного внимания. Описанные рекомендации — следующие шаги на пути от "система работает" к "система доказала, что работает надёжно".
